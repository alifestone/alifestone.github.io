# 個人部落格建置計畫書
### 使用 Claude Design 設計、部署於 GitHub Pages

---

## 一、專案目標

打造一個具備個人風格（含爬行邊框的 chibi 動畫角色）的技術部落格，架設於 GitHub Pages，用來記錄機器人研究（IDLab / LeRobot / SmolVLA）、競程解題、資安 CTF、日文學習等內容。整體流程分為「設計」與「工程」兩條線，透過 Claude Design 產出視覺與原型，再交接給程式碼實作，最終部署上線。

---

## 二、工具角色分工

| 階段 | 使用工具 | 說明 |
|---|---|---|
| 需求釐清、資訊架構 | Claude（聊天） | 討論版型、內容分類、技術選型 |
| 視覺設計、原型 | **Claude Design** | 建立設計系統、頁面原型、互動預覽 |
| 程式碼實作 | Claude Code（或手動） | 將原型轉換為可部署的靜態網站程式碼 |
| 版本控制與部署 | GitHub + GitHub Pages | 儲存原始碼、自動化建置與發佈 |
| 動畫素材 | Python（Pillow 等）+ Claude Design | 產生 SVG/Sprite Sheet，並在原型中預覽動畫效果 |

---

## 三、完整 Workflow

### 階段 0：前置準備（半天內）
1. 確認網域策略：使用 `<username>.github.io`（個人根網站）或 `<username>.github.io/blog-name`（專案網站）。
2. 決定技術路線：
   - **純靜態 HTML/CSS/JS**（適合 Claude Design 直接匯出，最貼近你現有的 chibi 動畫實驗）
   - 或 **靜態網站產生器**（如 Astro、Hugo、Jekyll — GitHub Pages 原生支援 Jekyll）
   - 建議：初期用純靜態 HTML 起步，之後若文章量變多再遷移到 Astro/Hugo。
3. 在 GitHub 建立 repo（例如 `blog` 或 `<username>.github.io`）。

### 階段 1：需求與資訊架構（與 Claude 對談）
1. 列出網站需具備的頁面：首頁、文章列表、單篇文章頁、關於我、專案/作品集頁。
2. 決定內容分類（例如：機器人研究、競程、資安、日文/歷史隨筆）。
3. 決定風格關鍵字（例如：可愛但技術感、深色系、像素/手繪風角色點綴），這些關鍵字會直接餵給 Claude Design 當 prompt 材料。
4. 輸出：一份簡短的「網站規格摘要」（頁面清單 + 風格描述 + 內容分類），作為下一階段的 prompt 依據。

### 階段 2：用 Claude Design 建立設計系統與原型
前往 `claude.ai/design` 建立新專案，依序進行：

1. **建立專案並匯入素材**
   - 若已有 chibi 角色 SVG / sprite sheet，直接上傳，讓 Claude Design 依此延伸配色與風格。
   - 若還沒有既有網站，可省略「匯入既有網站設計系統」這步，直接用文字描述從零開始。

2. **描述需求，產出首版設計**
   - 提示內容包含：目標（技術部落格首頁）、版面（導覽列、文章卡片列表、側邊欄或頁尾）、內容（分類標籤、最新文章、關於我簡介）、風格（深色主題、chibi 角色沿邊框爬行的裝飾動畫）。
   - Claude Design 會在畫布上產生第一版設計。

3. **迭代精修**
   - 用「聊天」做大範圍調整（例如「配色改成更冷色調」）。
   - 用「畫布上的行內註解」做局部調整（例如點選某按鈕說「這裡改成圓角」）。
   - 用「調整滑桿」微調間距、字級、顏色。
   - 針對動畫角色，可請 Claude Design 產出多個頁面（首頁、文章頁）分別預覽角色在不同版面下的爬行路徑效果。

4. **產出多個頁面模板**
   - 首頁（文章列表）
   - 文章內頁（含程式碼區塊樣式、圖片、目錄）
   - 關於我 / 作品集頁
   - 404 頁面（可以放 chibi 角色的小彩蛋動畫）

5. **確認可及性與響應式**
   - 請 Claude Design 檢查對比度、行動裝置版面是否合理。

### 階段 3：匯出與交接
1. 在 Claude Design 專案右上角使用「匯出」功能：
   - 匯出為 **HTML**（最適合直接進到靜態網站程式碼）。
   - 或直接「交接給 Claude Code」，讓 Claude Code 接手把設計轉成可維護的專案結構（HTML/CSS/JS 或框架化的元件）。
2. 若使用 Claude Code 交接，可在終端機執行 `/design-sync` 之類指令讓設計系統與程式碼保持同步（適合之後持續迭代版型時使用）。

### 階段 4：工程化與整合動畫
1. 建立專案資料夾結構，例如：
   ```
   blog/
     index.html
     posts/
       2026-07-11-first-post.html
     assets/
       css/
       js/
       sprites/chibi_walk.svg
     _posts/ (若改用 Jekyll)
   ```
2. 把 Claude Design 匯出的 HTML/CSS 落地為實際檔案，並把既有的 chibi sprite sheet 動畫（CSS animation 或 JS 控制 SVG 位移）整合進頁首/頁尾邊框。
3. 若文章數量會持續增加，考慮：
   - 用 Jekyll（GitHub Pages 原生支援，Markdown 寫文章即可自動產生頁面）
   - 或用 Astro/Hugo（本地 build 後把產出的靜態檔案推上 `gh-pages` 分支或 `docs/` 資料夾）

### 階段 5：部署到 GitHub Pages
1. 將程式碼推送到 GitHub repo。
2. 到 repo 的 **Settings → Pages**：
   - 若是純靜態 HTML：選擇 branch（如 `main`）與資料夾（`/root` 或 `/docs`）。
   - 若用 Jekyll：GitHub 會自動偵測並建置，不需額外設定。
   - 若用 Astro/Hugo 等需要 build 的框架：建議搭配 **GitHub Actions** 自動化建置與部署（見下方 CI/CD）。
3. 確認自訂網域（如有）並設定 `CNAME`。

### 階段 6：CI/CD 自動化（建議）
使用 GitHub Actions，於 push 到 `main` 時自動 build 並部署到 `gh-pages` 分支，範例流程：
1. `on: push to main`
2. 安裝相依套件（若用 Astro/Hugo）
3. 執行 build 指令
4. 使用 `peaceiris/actions-gh-pages` 或官方 `actions/deploy-pages` 部署產出物

好處：之後只要 `git push`，不用手動搬檔案，設計端（Claude Design）與工程端（GitHub Actions）完全解耦。

### 階段 7：內容經營與迭代
1. 建立文章模板（Markdown 或 HTML），固定 frontmatter（標題、日期、分類、標籤）。
2. 每次新增大改版視覺，回到 Claude Design 專案繼續迭代（設計系統會延續，不用從零開始）。
3. 定期檢查：
   - Lighthouse 效能與可及性分數
   - RWD 在手機上的呈現
   - chibi 動畫在不同瀏覽器的效能（避免過度重繪）

## 四、時間規劃建議（可依你課業/實驗室時程調整）

| 週次 | 任務 |
|---|---|
| Week 1 | 階段 0–1：確定架構與規格摘要 |
| Week 2 | 階段 2：Claude Design 出稿與迭代 |
| Week 3 | 階段 3–4：匯出、落地程式碼、整合 chibi 動畫 |
| Week 4 | 階段 5–6：部署 GitHub Pages、設定 CI/CD |
| 之後 | 階段 7：持續產文與版型迭代 |

## 五、待你確認的關鍵決策

1. 網站要用純靜態 HTML，還是導入 Jekyll/Astro 等產生器？

3. Claude Design 匯出後，是要自己手刻程式碼，還是交給 Claude Code 接手？
